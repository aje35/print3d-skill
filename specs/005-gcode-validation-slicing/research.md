# Research: G-code Validation & Slicing

## R1: G-code Parsing Approach

**Decision**: Custom streaming line-by-line parser with regex extraction, no external G-code library.

**Rationale**: G-code is a simple line-oriented format (one command per line). The parsing task is specialized — we need slicer metadata extraction from comments, not full motion planning. Existing Python libraries (pygcode, gcodeparser) are either unmaintained, overly complex for our needs, or lack slicer-specific comment parsing. A custom parser gives us full control over slicer-specific comment formats and keeps dependencies minimal (core tier = pip-only).

**Alternatives considered**:
- `pygcode`: Full G-code parser with motion semantics. Overkill — we need parameter extraction, not toolpath simulation. Also unmaintained.
- `gcodeparser`: Lightweight but no slicer comment parsing. Would still need custom code for the most valuable part (metadata extraction).
- Line-by-line with `re` module: Chosen. Simple, fast, zero dependencies. Pre-compiled regex patterns for each command type.

**Performance**: Streaming line-by-line with `open()` is memory-efficient for large files. Pre-compiled regex patterns avoid re-compilation per line. For a 100MB file (~2-3M lines), expect parsing in 2-4 seconds on modern hardware.

## R2: Slicer Comment Format Patterns

**Decision**: Implement per-slicer comment parsers behind a common interface, with auto-detection of slicer origin.

**Rationale**: Each slicer embeds metadata in different comment formats. Auto-detection avoids requiring users to specify the slicer. The four major slicers cover >95% of FDM G-code files.

**Slicer identification patterns**:
- **PrusaSlicer**: `; generated by PrusaSlicer` in file header
- **Bambu Studio**: `; generated by BambuStudio` or `; BambuStudio` in header
- **OrcaSlicer**: `; generated by OrcaSlicer` in header (fork of Bambu Studio)
- **Cura**: `; Generated with Cura_SteamEngine` or `;FLAVOR:` in header

**Metadata extraction patterns by slicer**:

| Parameter | PrusaSlicer | Bambu/OrcaSlicer | Cura |
|-----------|-------------|-------------------|------|
| Print time | `; estimated printing time (normal mode) = Xh Xm Xs` | `; total estimated time: X` | `;TIME:X` (seconds) |
| Filament used | `; filament used [mm] = X` | `; filament used [mm] = X` | `;Filament used: Xm` |
| Layer height | `; layer_height = X` | `; layer_height = X` | `;Layer height: X` |
| Layer change | `; LAYER_CHANGE` or `; Z:X` | `; CHANGE_LAYER` | `;LAYER:N` |
| Type markers | `;TYPE:Perimeter` | `;TYPE:Outer wall` | `;TYPE:WALL-OUTER` |
| Nozzle temp | `; nozzle_temperature = X` | `; nozzle_temperature = X` | Embedded in `M104`/`M109` |
| Bed temp | `; bed_temperature = X` | `; bed_temperature = X` | Embedded in `M140`/`M190` |
| Retraction | `; retract_length = X` | `; retraction_length = X` | `;SETTING_3` encoded |

**Fallback**: If slicer cannot be identified, extract parameters from G-code commands only (M104, M109, M140, M190, G1 F, etc.). Comment-based metadata will be empty but core parameters still available.

## R3: Slicer CLI Integration

**Decision**: Wrap PrusaSlicer and OrcaSlicer CLIs via subprocess, using their native profile systems.

**PrusaSlicer CLI**:
- Binary: `prusa-slicer` (macOS/Linux), `prusa-slicer-console.exe` (Windows)
- Slice command: `prusa-slicer --export-gcode -o output.gcode input.stl`
- Profile loading: `--load profile.ini` or named profiles from config dir
- Setting overrides: `--layer-height 0.3 --fill-density 30%`
- Version check: `prusa-slicer --version`
- Config dir: `~/.config/PrusaSlicer/` (Linux), `~/Library/Application Support/PrusaSlicer/` (macOS)

**OrcaSlicer CLI**:
- Binary: `orca-slicer` (fork of BambuStudio, which forked PrusaSlicer)
- CLI interface: Similar to PrusaSlicer but with some Bambu-specific extensions
- Slice command: `orca-slicer --export-gcode -o output.gcode input.stl`
- Config dir: `~/.config/OrcaSlicer/` (Linux), `~/Library/Application Support/OrcaSlicer/` (macOS)

**Profile strategy**: Use the slicer's own installed profiles by name, not our internal knowledge profiles. Our profiles are for validation; the slicer's profiles drive slicing. Users reference slicer profiles by name (e.g., "0.20mm QUALITY"), and can override individual settings.

**Alternatives considered**:
- CuraEngine CLI: Possible future addition but different CLI interface (uses JSON configs). Deferred.
- Translating internal profiles to slicer configs: Far too complex — slicer profiles have hundreds of settings. Better to use native profiles.

## R4: Printer API Protocols

**Decision**: Synchronous HTTP/MQTT clients using `requests` (OctoPrint, Moonraker) and `paho-mqtt` (Bambu Lab), behind a common PrinterBackend interface.

**OctoPrint REST API**:
- Default: `http://<host>:5000/api/`
- Auth: `X-Api-Key` header
- Status: `GET /api/printer` (state, temps, flags)
- Upload: `POST /api/files/local` (multipart form with G-code file)
- Print: `POST /api/files/local/<filename>` with `{"command": "select", "print": true}`
- Python: `requests` library (no official client needed — simple REST)

**Moonraker/Klipper REST API**:
- Default: `http://<host>:7125/`
- Auth: API key or trusted client (IP-based)
- Status: `GET /printer/objects/query?print_stats&heater_bed&extruder`
- Upload: `POST /server/files/upload` (multipart)
- Print: `POST /printer/print/start?filename=<name>`
- Python: `requests` library

**Bambu Lab MQTT**:
- Protocol: MQTT over TLS (port 8883)
- Auth: Device serial + access code
- Status: Subscribe to `device/<serial>/report`
- Print: Publish to `device/<serial>/request` with job payload
- Python: `paho-mqtt` library (lightweight, well-maintained)
- Note: File transfer uses FTP to the printer's SD, then MQTT to start

**Common interface**: `PrinterBackend` ABC with methods: `connect()`, `status()`, `upload(gcode_path)`, `start_print(filename)`, `disconnect()`. Each API gets a concrete implementation.

**Alternatives considered**:
- asyncio: Adds complexity for a library that primarily does one-shot operations. Synchronous is simpler and sufficient since we're not building a long-running server.
- `pybambu`: Existing library for Bambu Lab, but adds a heavy dependency and may not be stable enough. Better to use `paho-mqtt` directly with minimal protocol code.

## R5: Printer Connection Configuration

**Decision**: YAML configuration file at a well-known path, following the same structured data pattern as the knowledge system.

**Configuration path**: `~/.config/print3d-skill/printers.yaml` (XDG on Linux), `~/Library/Application Support/print3d-skill/printers.yaml` (macOS)

**Format**:
```yaml
printers:
  - name: "My Bambu X1C"
    type: bambu
    host: 192.168.1.100
    serial: "01P00A000000000"
    access_code: "12345678"
  - name: "Workshop Prusa"
    type: octoprint
    host: octopi.local
    port: 5000
    api_key: "ABCDEF123456"
  - name: "Voron 2.4"
    type: moonraker
    host: voron.local
    port: 7125
```

**Rationale**: YAML is consistent with the knowledge system. A dedicated config file keeps credentials separate from project files (never committed to git). The path follows OS conventions.

**Alternatives considered**:
- Environment variables: Good for CI but poor UX for multiple printers.
- Per-call parameters: Too verbose for repeated use.
- JSON config: YAML is more readable for humans editing printer configs.

## R6: Validation Severity Boundaries

**Decision**: Two-tier severity with clear decision rules.

**Fail** (blocks printing):
- Print dimensions exceed build volume
- Hotend temperature exceeds printer's max temperature
- Bed temperature exceeds printer's max bed temperature
- Required features not supported (e.g., heated bed needed but printer has none)

**Warn** (advisory, allows printing with acknowledgment):
- Temperatures outside material's recommended range (but within printer capability)
- Speeds outside recommended range for material
- Retraction settings mismatched for extruder type
- First layer settings outside best-practice ranges
- Print time exceeds threshold (default 72h)
- Filament usage seems excessive for model size

**Pass**: All checks within expected ranges.

**Rationale**: "Fail" means the print will physically fail or is unsafe. "Warn" means the print may have quality issues but won't damage equipment. This distinction prevents unnecessary blocking while catching dangerous misconfigurations.

## R7: Core vs Extended Tier Mapping

**Decision**: G-code parsing and settings validation are core tier. Slicer CLI and printer control are extended tier.

| Component | Tier | Dependencies |
|-----------|------|-------------|
| G-code parser | Core | None (Python stdlib: re, io) |
| Settings validator | Core | None (uses knowledge YAML files) |
| Material/printer profiles | Core | PyYAML (already a dependency) |
| Validate mode handler | Core | Uses parser + validator |
| PrusaSlicer CLI wrapper | Extended | PrusaSlicer binary |
| OrcaSlicer CLI wrapper | Extended | OrcaSlicer binary |
| OctoPrint backend | Extended | requests (pip) + OctoPrint server |
| Moonraker backend | Extended | requests (pip) + Moonraker server |
| Bambu Lab backend | Extended | paho-mqtt (pip) + Bambu printer |

**Rationale**: Constitution VI requires core tier to be pip-installable. The parser and validator use only Python stdlib + existing dependencies (PyYAML). Extended features add new pip dependencies (requests, paho-mqtt) but these are lightweight. The `requests` library is already widely used and likely an indirect dependency. `paho-mqtt` is the only truly new dependency.
